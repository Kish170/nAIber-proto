# RAG Pipeline Audit Report — 2026-04-24

## Executive Summary

The RAG pipeline is **non-functional for vector search**. Qdrant returns zero results for every retrieval query due to a payload structure mismatch between how LangChain stores data (`metadata.userId`) and how the filter queries it (`userId` at root level). All retrieval is falling back to KG-only discovery, which returns low-relevance results (finalScores 0.03-0.19) because the reranking formula gives KG a 0.3 weight while Qdrant gets 0.7 — but Qdrant always contributes 0.0. The post-call population pipeline works end-to-end and produces reasonable data, but NER extracts generic entity names (e.g., "User's Granddaughter" instead of the actual name), all graph scores are hardcoded to 1.0, and LangSmith traces are not landing in the expected projects.

**Impact**: The AI companion cannot recall any specific memories from previous conversations via vector search. It only surfaces vaguely related highlights discovered through the knowledge graph, with very low confidence scores. The user experience is that the companion doesn't remember anything meaningful.

## 1. Data Quality

### 1.1 Qdrant (Vector Store)

**Collection**: `nAIber-proto` — 194 total points, 1536-dimension vectors (text-embedding-3-small), cosine distance.

| Metric | Value | Assessment |
|---|---|---|
| Points for test user (`e4742847-...`) via `userId` filter | 0 | BROKEN |
| Points for test user via `metadata.userId` (nested) | 10 | Data exists but invisible to retrieval |
| Total points in collection | 194 | Across 3 users |
| `indexed_vectors_count` | 0 | Normal for <10k points (brute-force scan) |

**Root Cause — Payload Structure Mismatch**:

Two incompatible payload formats exist in the same collection:

```
Format A (65 points, old seed script):
  { userId: "...", conversationId: "...", highlight: "..." }

Format B (129 points, LangChain QdrantVectorStore):
  { content: "...", metadata: { userId: "...", conversationId: "...", ... } }
```

Both `VectorStoreClient` implementations (llm-server write, mcp-server read) use the filter `{ key: "userId", match: { value: userId } }`, which only matches Format A (root-level `userId`). All 10 points for the test user are Format B (nested under `metadata`), so the filter returns zero results.

Additionally, Qdrant Cloud requires a payload index to filter on nested keys (`metadata.userId`), and no such index exists.

**Embedding Storage**: The post-call pipeline reported storing 4 new embeddings with specific point IDs (`08dd4f05-...`, `1ab640c9-...`, etc.), but these IDs do not exist in the collection. The `addVectors` call via LangChain QdrantVectorStore either failed silently or wrote to a different location. No errors appeared in the llm-server logs.

**Text Quality**: The 4 highlights stored from the test call had text lengths of 25, 33, 45, and 51 characters:
- "lunch with daughter Sarah" (25 chars)
- "neighbor Tom helped in the garden" (33 chars)
- "positive blood pressure update from Dr. Patel" (45 chars)
- "husband Rob's stable condition after hospital visit" (51 chars)

These are short but semantically meaningful. However, they are summary-derived highlights, not direct quotes — the GPT summary prompt determines their quality.

## 2. Post-Call Pipeline Quality

### 2.1 Transcript Retrieval

The first attempt to fetch the transcript failed: `"Unable to retrieve transcript"`. BullMQ retried and succeeded on attempt 2. This is expected behavior — ElevenLabs transcript availability has a delay after call end. The 3-second delay before job dispatch may be insufficient.

### 2.2 Summary Generation

The GPT summary produced:

- **topicsDiscussed**: 4 topics — "family lunch", "gardening", "health updates", "husband's hospital visit" — all meaningful and specific
- **keyHighlights**: 4 highlights — short but semantic

Assessment: Summary generation quality is **good**. The prompt produces specific, meaningful outputs.

### 2.3 Topic Matching

All 4 topics from the test call were created as new (0 matched existing). The cosine threshold of 0.78 for matching is reasonable. However:
- Topics like "gardening" and "birdwatching" (existing) may be semantically close enough to link but didn't match — suggesting the threshold could be slightly aggressive
- No existing topics were matched at all, which could indicate the test user had sparse topic data before this call

### 2.4 Embedding Storage

4 embeddings reported as stored successfully. However, as detailed in Section 1.1, these point IDs do not exist in Qdrant when queried directly. The `addVectors` call appears to complete without error but the data is not persisted or not accessible. This is the most critical pipeline failure.

### 2.5 NER (Named Entity Recognition)

GPT-4o extracted 5 persons from the transcript:

| Name | Role | Assessment |
|---|---|---|
| Sarah | daughter | Correct |
| Tom | neighbor | Correct |
| Dr. Patel | doctor | Correct |
| Rob | husband | Correct |
| User's Granddaughter | granddaughter | **Generic** — not an actual name |

The NER prompt does not instruct the model to skip entities without a proper name. When a person is mentioned by role only ("my granddaughter"), the model fabricates a label like "User's Granddaughter" and stores it as if it were a name. This creates noise in the KG — 43% of all persons in Neo4j are these generic labels.

Additionally:
- No confidence scoring on extractions — all entities are treated equally
- `highlightIndices` is always empty `[]` — this field appears unused
- Person IDs are SHA256 hashes of `name:role`, so "User's Granddaughter:granddaughter" and a future properly-named granddaughter would be separate nodes

## 3. Retrieval Quality

### 3.1 Vector Search: Completely Broken

Across 8 retrieval calls during the test call, Qdrant returned **zero results every time**:

| Query | Raw Docs | Filtered Docs | Source |
|---|---|---|---|
| "baking" | 0 | 0 | 100% KG |
| "family names" | 0 | 0 | 100% KG |
| "gingerbread cookies" | 0 | 0 | 100% KG |
| "technical issues" | 0 | 0 | 100% KG |
| "garden" | 0 | 0 | 100% KG |
| "husband's name" | 0 | 0 | 100% KG |
| "Rob" | 0 | 0 | 100% KG |

Root cause: `userId` filter mismatch (see Section 1.1).

### 3.2 KG Discovery: Low Relevance

With Qdrant returning nothing, retrieval falls back entirely to KG discovery. The pipeline:
1. Queries Postgres for all user topics (13 topics at retrieval time)
2. Computes cosine similarity between query embedding and each topic embedding
3. Takes top 5 topics
4. Fetches highlights linked to those topics from Neo4j
5. Computes `kgScore` as the product of topic similarity and highlight importance (always 1.0)
6. Final score = `0.7 * qdrantScore + 0.3 * kgScore` = `0.3 * kgScore` (since qdrant = 0)

**Score distribution across all 8 queries**:

| Metric | Value |
|---|---|
| Max finalScore | 0.192 |
| Min finalScore | 0.040 |
| Mean finalScore | 0.068 |
| Scores above 0.1 | 11% |
| Scores above 0.15 | 3% |

These scores are extremely low. The alpha=0.7 weighting for Qdrant means KG-only results are capped at 0.3 max even with perfect topic matching. This makes KG discovery structurally unable to produce high-confidence results without Qdrant support.

**Recurring highlight IDs**: The same 5-6 highlight IDs (`664cfafb`, `43cc08fe`, `56c7cf91`, `cce512de`, `bdca9d1f`, `12e04b02`) appear across nearly every query regardless of the topic. This suggests the highlight pool is small enough that the same highlights are returned for every query, providing no differentiation.

### 3.3 Topic Matching Quality

Some topic matches during retrieval were reasonable:
- "gingerbread cookies" → top match: `gingerbread cookies (1.000)`, `baking with granddaughter (0.494)`, `baking (0.488)` — good
- "husband's name" → top match: `family names (0.423)` — reasonable
- "garden" → top match: `birdwatching (0.264)` — **poor**, "gardening" wasn't in the topic list yet (created after this retrieval, during post-call)

Note: Retrieval during the call uses pre-existing topics only. Topics created by the current call's post-call pipeline are not available until the next call.

### 3.4 KG Enrichment (Stream 1)

The KG enrichment stream (`enrichQdrantResults`) was never exercised because Qdrant returned zero results every time: `"enrichQdrantResults: no point IDs to enrich"`. This entire code path is untested in production.

## 6. Threshold and Configuration Issues

### 6.1 RAG_MEMORY_SIMILARITY_THRESHOLD (0.45)

Currently irrelevant because Qdrant returns zero results. Once the filter bug is fixed, 0.45 is a reasonable starting threshold for text-embedding-3-small cosine similarity. Needs re-evaluation with working vector search.

### 6.2 Topic Matching Threshold (0.78)

Used in `GeneralPostCallGraph.matchTopics()` to decide whether a new topic matches an existing one. At 0.78, this is moderately strict. Consider:
- "gardening" vs "garden" would likely exceed 0.78 (good)
- "health updates" vs "arthritis pain" would likely fall below 0.78 (may create duplicates)

The 5 Postgres topics missing from Neo4j suggest some topic fragmentation.

### 6.3 Highlight-Topic Linking Threshold (0.4)

Used in `KGPopulationService.populateRelationships()`. Only 5 of 68 possible links passed (7.3%). This is selective but may be appropriate — linking every highlight to every topic would create noise.

### 6.4 Alpha (Qdrant/KG Weight: 0.7/0.3)

The `alpha=0.7` weighting heavily favors Qdrant. When Qdrant is non-functional, KG results are structurally capped at `0.3 * kgScore`, making even perfect KG matches appear low-confidence. If KG is intended as a fallback, this weighting should be dynamic based on whether Qdrant returned results.

### 6.5 pgTopicLimit (5)

During KG discovery, only the top 5 most-similar Postgres topics are used. With 13-17 topics, this means ~30-40% of topics are considered. This is reasonable for performance but could miss relevant topics ranked 6th or lower.

## 7. LangSmith Observability

**Status**: LangSmith traces are **not landing** in the expected projects.

- Only the `default` project exists in LangSmith (created 2025-07-05)
- No `naiber-llm-postcall` or `naiber-mcp-retrieval` projects exist
- The `LANGCHAIN_PROJECT` env vars are set on both containers but traces appear to either:
  - Not be sent at all (traceable wrappers may not auto-create projects)
  - Be sent to the default project (not verified)
  - Be blocked by a network/auth issue from within Docker containers

The `audit-langsmith-traces.ts` script fails with `Project[name=naiber-llm-postcall] not found`.

**Impact**: The LangSmith instrumentation added in Phase 1 is not producing visible traces. The console.log instrumentation works correctly and provided all the data for this audit, but structured LangSmith traces with span trees, latency metrics, and cost tracking are not available.

**Possible causes**:
1. LangSmith API key may lack project creation permissions
2. `traceable()` wrappers require the project to exist before sending traces
3. Docker container DNS/networking may block outbound HTTPS to `api.smith.langchain.com`
4. The `langsmith` SDK version may require explicit initialization beyond env vars

## 8. Known Structural Issues (Code Review)

These were identified during instrumentation and confirmed during the audit:

| Issue | Location | Impact |
|---|---|---|
| `importanceScore` hardcoded to 1.0 | `KGPopulationService.populateRelationships()` | No quality differentiation between highlights |
| `RELATED_TO` strength hardcoded to 1.0 | `KGPopulationService.populateRelationships()` | Topic-topic edges all equal weight |
| `deriveInterestedInEdges` never called | `GraphRepository.ts` (exists) / `KGPopulationService.ts` (never calls it) | User→Topic INTERESTED_IN edges never created (0 in Neo4j) |
| Embedding averaging for topic updates | `GeneralPostCallGraph.updateTopics()` | Repeated averaging of topic embeddings risks semantic drift from the topic's actual meaning |
| `source` was stripped from output | `retrieveMemoriesHandler` (now fixed by instrumentation) | LLM previously had no visibility into where memories came from |
| `VectorStoreClient` userId filter mismatch | Both `apps/llm-server` and `apps/mcp-server` | Root cause of Qdrant retrieval failure |

## 10. Recommended Next Steps

1. **Fix P0 immediately**: The Qdrant filter mismatch makes the entire vector retrieval non-functional. Fix the filter key, verify data accessibility, then re-run this audit to establish a working baseline.

2. **Investigate embedding persistence**: The 4 test-call embeddings reported as stored but not found in Qdrant is a separate issue from the filter. Debug the `addVectors` call path — add try/catch with explicit error logging, then verify point existence after write.

3. **Fix NER prompt**: Quick win to eliminate generic entity noise. Update the system prompt in `NERService.ts`.

4. **Make alpha dynamic**: Simple conditional in `mergeAndDeduplicate()` — if no Qdrant results, set alpha to 0 so KG scores are used at full weight.

5. **Debug LangSmith**: Once traces are landing, re-run the evaluation dataset (`scripts/langsmith/evaluate-retrieval.ts`) to get structured quality metrics.

6. **Re-run full audit after P0 fixes**: With working vector search, the retrieval quality picture will look very different. The threshold tuning, score distribution, and source diversity assessments in this report are all based on a broken pipeline and will need reassessment.
