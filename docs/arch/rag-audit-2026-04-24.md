3. Adding post-write verification in `addMemoriesWithIds` (retrieves IDs after upsert, throws if any missing)

## 2. Post-Call Pipeline Quality

### 2.1 Transcript Retrieval

Delay increased from 3s → 6s. Transcript fetch now succeeds on the first attempt in recent calls. BullMQ retry-on-failure remains in place as a safety net.

### 2.2 Summary Generation

Recent calls produce high-quality summaries:
- **summaryText**: 200–500 chars — concise and informative
- **topicsDiscussed**: Short general labels (1–3 words) per the updated prompt — e.g., "gardening", "baking", "family visit" — designed for cross-call reuse
- **keyHighlights**: 1–4 highlights per call, avg 40 chars

Topic label design was improved: the prompt now instructs the model to use short general category labels and blocks meta-topics like "end of conversation", "general chat", etc.

### 2.3 Topic Matching

With short general labels, more topics now match across calls (cosine threshold 0.78 for `text-embedding-3-small`). Topic proliferation is reduced compared to the initial audit.

### 2.4 Embedding Storage

**Status: Working.** 16 vectors stored with 100% metadata completeness. Post-write verification confirms all point IDs exist in Qdrant immediately after upsert. No silent failures observed.

Topic embeddings are now **replaced** (not averaged) on each update — eliminates embedding drift from repeated averaging.

### 2.5 NER (Named Entity Recognition)

**Improved.** Updated prompt:
- Only extracts persons referred to by name
- Requires a verbatim `quote` from the transcript containing the name
- Post-filter drops any entry where the name isn't found in the provided quote
- Good/bad examples added to the prompt to illustrate the distinction

Pre-fix generic rate: ~43% of KG person nodes were "User's X"-style labels. Post-fix: these entries no longer pass the quote verification step.

### 2.6 KG Population

`deriveInterestedInEdges` is now called at the end of `populateRelationships()`. User→Topic INTERESTED_IN edges are created when a topic's MENTIONS count reaches 3. With 8 conversations accumulated, some topics are approaching or have exceeded this threshold.

## 3. Retrieval Quality

### 3.1 Vector Search: Now Functional

Qdrant returns results during calls. From mcp-server logs during the most recent test call:
```
[VectorStoreClient] searchByEmbedding userId=e4742... k=5 results=5 scores=[0.6762, 0.4419, ...]
[retrieveMemories] rawDocs=5 filteredDocs=2 threshold=0.45
```

Source breakdown for that call: `qdrant=1, kg_discovery=1, both=0`.

### 3.2 Representative Query Recall (Embedding Audit)

| Query | Top Score | Above Threshold (0.45) | Assessment |
|---|---|---|---|
| "Tell me about family names" | 0.5228 | 1/5 | OK |
| "How have you been feeling?" | 0.2709 | 0/5 | Low |
| "quantum computing research" | 0.1901 | 0/5 | Expected (irrelevant) |
| "What did we talk about last time?" | 0.3285 | 0/5 | Low |

**Observation**: Specific-topic queries (matching actual stored highlight text) return above-threshold results. General/open-ended queries ("How have you been feeling?") return low scores across all highlights. This is a consequence of short, event-specific highlights — they match well when the query names the same event but poorly when the query is generic.

This is a known tradeoff of the current highlight extraction strategy. Highlights like "mentioned things going well in the kitchen" are concrete but don't match a general wellbeing query. Not a bug, but a content quality consideration for future iterations.

### 3.3 KG Discovery: Dynamic Alpha Working

With `effectiveAlpha = qdrantDocuments.length > 0 ? config.alpha : 0.0`:
- When Qdrant returns results: `alpha=0.7` (Qdrant-weighted), KG enriches
- When Qdrant returns nothing: `alpha=0.0`, KG scores use full weight instead of being capped at 0.3

KG-only fallback now produces usable finalScores instead of the structurally degraded 0.03–0.19 range from pre-fix.

### 3.4 Near-Duplicate Analysis

0 near-duplicate clusters (cosine > 0.95) across all 16 vectors. Each highlight is semantically distinct.

## 4. Topic Embedding Drift Analysis

| Topic | Drift | Notes |
|---|---|---|
| Gardening | 0.9715 | Seed data — no real embedding stored |
| Family time | 1.0135 | Seed data — no real embedding stored |
| Arthritis pain | 0.9567 | Seed data — no real embedding stored |
| Baking | 1.0246 | Seed data — no real embedding stored |
| Social activities | 1.0221 | Seed data — no real embedding stored |

Drift values >1.0 indicate NULL or zero-vector stored embeddings — these topics were inserted by the seed script without real embeddings. Topics created by the live pipeline have proper embeddings and show no drift.

**Impact**: These 5 seed topics participate in KG discovery but their cosine similarity computations are unreliable (zero-vector dot products). They are Postgres-only (not in Neo4j), so they don't affect KG retrieval paths, only the Postgres-side topic embedding lookup in `KGRetrievalService`.

**Mitigation**: These topics could be re-embedded by running `embService.generateEmbedding(topic.topicName)` and updating the `topicEmbedding` field. Low priority since real call data is now accumulating.
