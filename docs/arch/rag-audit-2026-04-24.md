# RAG Pipeline Audit Report — 2026-04-24

## Executive Summary

**Re-audited 2026-04-26 after MCP tools + RAG expansion batch.**

New pipeline features shipped: 10 MCP tools (profile, relationship, event, and call-end tooling), RELATED_TO lateral expansion in KGRetrievalService (highlights and persons for related topics, with `expansionSource` tracking), topic/person/event Qdrant embeddings in post-call pipeline, and summary validation. Audit confirms all three stores are at 100% mutual consistency across 17 conversations. RELATED_TO co-occurrence formula is active and generating eligible expansion pairs. Seven INTERESTED_IN edges now derive from accumulated topic data.

**Outstanding items**: importanceScore scale inconsistency (some calls stored 0.7–0.9 instead of 7–9); person Qdrant entries missing `personId`/`name` metadata fields (fixed in pipeline, needs next call to verify); 7 Postgres topics without Qdrant vectors (4 pre-pipeline, 3 from calls before topic embedding was added); `call_events` migration pending deployment to Supabase (flagCallEvent tool functional, table not yet created).

**Previous state (2026-04-25 after P5–P8)**: 17 vectors, one richer highlight, avg score 0.3628, RELATED_TO and INTERESTED_IN edges active for new data only.

**Previous state (2026-04-24 after P0–P4)**: vector search functional, 16 highlights, 83% cross-store consistency. Short highlights (avg 40 chars), importanceScore hardcoded to 1.0, INTERESTED_IN edges unused, RELATED_TO strength hardcoded to 1.0.

**Original pre-fix state**: vector search returned 0 results; embeddings silently unreadable; KG-only fallback capped at 0.3 weight; generic NER labels; `deriveInterestedInEdges` never called.

### importanceScore Quality (Neo4j) — ⚠ Scale Inconsistency Detected

| Score | Count | Source |
|---|---|---|
| 1 | 10 | Pre-P5 highlights (hardcoded) |

| 7 | 3 | Most recent calls — correct 1–10 scale ✓ |
| 8 | 2 | Most recent calls — correct 1–10 scale ✓ |

The 0.7–0.9 scores represent the same semantic value as 7–9 but on the wrong scale. This is a prompt compliance issue from intermediate calls — the LLM output decimal values (e.g. `0.8`) instead of integers (e.g. `8`). The `getSignificantHighlights()` query filters `importanceScore >= 7`, so the 0.7–0.9 highlights are silently excluded from significant event retrieval. The prompt has not been changed yet — this should be hardened with an explicit integer-only instruction.
