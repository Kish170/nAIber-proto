## 2. Post-Call Pipeline Quality

### 2.1 Transcript Retrieval

Delay increased from 3s ‚Üí 6s. Transcript fetch now succeeds on the first attempt in recent calls. BullMQ retry-on-failure remains in place as a safety net.

### 2.2 Summary Generation

Recent calls produce high-quality summaries:
- **summaryText**: 200‚Äì500 chars ‚Äî concise and informative
- **topicsDiscussed**: Short general labels (1‚Äì3 words) per the updated prompt ‚Äî e.g., "gardening", "baking", "family visit" ‚Äî designed for cross-call reuse
- **keyHighlights**: 1‚Äì4 highlights per call, avg 40 chars

Topic label design was improved: the prompt now instructs the model to use short general category labels and blocks meta-topics like "end of conversation", "general chat", etc.

### 2.3 Topic Matching

With short general labels, more topics now match across calls (cosine threshold 0.78 for `text-embedding-3-small`). Topic proliferation is reduced compared to the initial audit.

### 2.4 Embedding Storage

**Status: Working.** 16 vectors stored with 100% metadata completeness. Post-write verification confirms all point IDs exist in Qdrant immediately after upsert. No silent failures observed.

Topic embeddings are now **replaced** (not averaged) on each update ‚Äî eliminates embedding drift from repeated averaging.

### 2.5 NER (Named Entity Recognition)

**Improved.** Updated prompt:
- Only extracts persons referred to by name
- Requires a verbatim `quote` from the transcript containing the name
- Post-filter drops any entry where the name isn't found in the provided quote
- Good/bad examples added to the prompt to illustrate the distinction

Pre-fix generic rate: ~43% of KG person nodes were "User's X"-style labels. Post-fix: these entries no longer pass the quote verification step.

### 2.6 KG Population

`deriveInterestedInEdges` is now called at the end of `populateRelationships()`. User‚ÜíTopic INTERESTED_IN edges are created when a topic's MENTIONS count reaches 3. With 8 conversations accumulated, some topics are approaching or have exceeded this threshold.

**Observation**: Specific-topic queries (matching actual stored highlight text) return above-threshold results. General/open-ended queries ("How have you been feeling?") return low scores across all highlights. This is a consequence of short, event-specific highlights ‚Äî they match well when the query names the same event but poorly when the query is generic.

This is a known tradeoff of the current highlight extraction strategy. Highlights like "mentioned things going well in the kitchen" are concrete but don't match a general wellbeing query. Not a bug, but a content quality consideration for future iterations.

### 3.3 KG Discovery: Dynamic Alpha Working

- When Qdrant returns results: `alpha=0.7` (Qdrant-weighted), KG enriches
- When Qdrant returns nothing: `alpha=0.0`, KG scores use full weight instead of being capped at 0.3

KG-only fallback now produces usable finalScores instead of the structurally degraded 0.03‚Äì0.19 range from pre-fix.

Drift values >1.0 indicate NULL or zero-vector stored embeddings ‚Äî these topics were inserted by the seed script without real embeddings. Topics created by the live pipeline have proper embeddings and show no drift.

**Impact**: These 5 seed topics participate in KG discovery but their cosine similarity computations are unreliable (zero-vector dot products). They are Postgres-only (not in Neo4j), so they don't affect KG retrieval paths, only the Postgres-side topic embedding lookup in `KGRetrievalService`.

**Mitigation**: These topics could be re-embedded by running `embService.generateEmbedding(topic.topicName)` and updating the `topicEmbedding` field. Low priority since real call data is now accumulating.
